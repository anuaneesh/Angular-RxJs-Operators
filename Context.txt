[1:40 - 3:00] switchMap

Visual

Timeline animation

A -------- X

An ------- X

Ang ------ X

Angular -------- Complete

Previous requests disappear.

Only the last remains.

Voiceover

Think of switchMap as someone with a very short attention span.

The moment something new arrives...

It completely forgets what it was doing.

Every previous request gets cancelled.

Only the latest survives.

That's exactly what we want in search autocomplete.

Because imagine this...

The user types "Angular."

But the request for just "A" takes longer to complete.

Without switchMap...

Your UI first shows Angular results...

Then suddenly changes back to results for "A."

That's called a race condition.

switchMap prevents this by cancelling outdated requests.

Code Demo
this.searchControl.valueChanges.pipe(
  debounceTime(300),
  distinctUntilChanged(),
  switchMap(term =>
      this.http.get(`/api/search?q=${term}`)
  )
)

This one confused me the most.

Here's the easiest way to remember it.

"I'm busy."

Anything where duplicate requests would be disastrous.

[8:00 - 9:00] The Cheat Sheet

Beautiful animated table.

Question	Operator
Only latest matters	switchMap
Run everything	mergeMap
Order matters	concatMap
Ignore duplicates	exhaustMap

Voiceover

Whenever a new value arrives...

Ask yourself these four questions.

Do I only care about the latest value?

Use switchMap.

Should every request execute?

Use mergeMap.

Does order matter?

Use concatMap.

Should duplicate requests be ignored while one is already running?

Use exhaustMap.

That's literally all you need to remember.
